# SQL-backed Card Query Repository Plan

This plan is the implementation handoff for replacing the current SQLite Card Query repository. It records the design
decisions from the grilling session so another agent can implement without re-litigating the query semantics.

## Goal

Replace the current in-memory SQLite `CardQueryRepository` evaluator with a SQL-backed implementation that keeps the
public `query_cards` tool shape stable while pushing supported filtering, sorting, limiting, relationship matching, and
Collection quantity aggregation into SQLite.

The current evaluator loads broad Card Identity, legality, tag, hierarchy, and Collection row sets into TypeScript
before
filtering. That is the performance problem. The replacement should be one SQL-backed implementation, not a production
fallback or a parallel runtime path.

## Non-Goals

- Do not expose raw SQL, physical table names, joins, or SQL fragments to the agent.
- Do not change the top-level `query_cards` envelope shape except for adding the explicit relationship-scope operators
  described below.
- Do not remove currently documented Card Query features. Operators, queryables, includes, sorting, and limits should
  stay
  available.
- Do not treat accidental behavior from the old in-memory evaluator as authoritative. Where behavior is ambiguous, this
  plan and `docs/plans/card-query.md` are the source of truth.
- Do not rely on the developer's local `.data/mtg-agent.sqlite` for tests. Tests must build their own temporary migrated
  databases and insert fixture data.

## Current State

- `docs/adr/0013-sql-backed-card-query-repository.md` records the architecture decision.
- `docs/plans/card-query.md` contains the broader Card Query design and SQL-backed implementation shape.
- The SQLite schema now includes Card Query-oriented indexes generated by migration `0004_bright_speed.sql`.
- The current repository implementation lives in `packages/sqlite/src/card-query-repository.ts` and evaluates most
  filters
  in TypeScript after broad table loads.
- The public core port is `CardQueryRepository.queryCards(input): Promise<Result<CardQueryResult, CardQueryError>>`.

## Core Boundary Decision

Keep the core repository port and agent-facing tool shape stable. SQL pushdown is internal to the SQLite adapter.

Allowed public-language addition:

- `withTagging`
- `withCollectionCard`

These are Card Query operators, not raw SQL escape hatches. They are added to make relationship scoping explicit and
agent-legible.

## Relationship Scope Operators

### `withTagging`

Shape:

```json
{
  "op": "withTagging",
  "args": [
    {"op": "hasTagInHierarchy", "args": [{"property": "tag.id"}, "draw-tag-id"]}
  ]
}
```

The single child filter must contain only tag relationship predicates and boolean groups over those predicates.

Allowed inside `withTagging`:

- `tag.id`
- `tag.slug`
- `tag.label`
- `tag.alias`
- `tag.weight`
- `hasTagInHierarchy` over `tag.id`
- `and` / `or` over allowed tag predicates

Not allowed inside `withTagging`:

- `identity.*`
- `legality.*`
- `collection.*`
- `not`

Meaning: there exists at least one `Card Identity Tagging` row for the Card Identity that satisfies the child filter.

Use `withTagging` when a tag concept must be qualified by tag metadata on the same tagging row.

Example: strong draw, where the draw-related tagging itself must be strong:

```json
{
  "op": "withTagging",
  "args": [
    {
      "op": "and",
      "args": [
        {"op": "hasTagInHierarchy", "args": [{"property": "tag.id"}, "draw-tag-id"]},
        {"op": "=", "args": [{"property": "tag.weight"}, "strong"]}
      ]
    }
  ]
}
```

This must not match a card that has a weak draw tag and an unrelated strong ramp tag.

### Independent Multi-Tag Search

Normal multi-tag searches should use independent tag predicates, not one shared tagging scope.

Example: cards that are both draw-related and ramp-related:

```json
{
  "op": "and",
  "args": [
    {"op": "hasTagInHierarchy", "args": [{"property": "tag.id"}, "draw-tag-id"]},
    {"op": "hasTagInHierarchy", "args": [{"property": "tag.id"}, "ramp-tag-id"]}
  ]
}
```

Meaning: the Card Identity has a draw-related tagging and has a ramp-related tagging. They may be different tagging
rows.

This is a valid and important deck-building use case. The SQL-backed implementation must not collapse independent tag
identity predicates into one impossible same-row tag requirement.

### Direct Tag Metadata Outside `withTagging`

Keep `tag.weight` and other tag queryables available outside `withTagging` because they are documented features.
Outside `withTagging`, a tag predicate is an independent identity-level positive relationship search.

Example:

```json
{"op": "=", "args": [{"property": "tag.weight"}, "strong"]}
```

Meaning: the card has at least one direct Card Identity Tagging with weight `strong`.

Agents should prefer `withTagging` when combining tag identity and tag metadata into one same-tagging requirement.

### `withCollectionCard`

Shape:

```json
{
  "op": "withCollectionCard",
  "args": [
    {"op": "=", "args": [{"property": "collection.locationName"}, "Simic Ramp Control"]}
  ]
}
```

The single child filter must contain only Collection relationship predicates and boolean groups over those predicates.

Allowed inside `withCollectionCard`:

- `collection.quantity`
- `collection.locationName`
- `collection.locationType`
- `collection.finish`
- `collection.altered`
- `collection.misprint`
- `and` / `or` over allowed Collection predicates

Not allowed inside `withCollectionCard`:

- `identity.*`
- `legality.*`
- `tag.*`
- `hasTagInHierarchy`
- `not`

Meaning: the Card Identity has a matching Collection row scope satisfying the child filter. Despite the singular
operator
name, `collection.quantity` still aggregates over the scoped matching `collection_card` rows.

Example: foil copies in Simic Ramp Control:

```json
{
  "op": "withCollectionCard",
  "args": [
    {
      "op": "and",
      "args": [
        {"op": "=", "args": [{"property": "collection.locationName"}, "Simic Ramp Control"]},
        {"op": "=", "args": [{"property": "collection.finish"}, "foil"]}
      ]
    }
  ]
}
```

### Collection Shorthand

Keep existing simple Collection predicates as ergonomic shorthand. A flat branch like this still scopes the same
matching
Collection rows:

```json
{
  "op": "and",
  "args": [
    {"op": "=", "args": [{"property": "collection.locationName"}, "Simic Ramp Control"]},
    {"op": "=", "args": [{"property": "collection.finish"}, "foil"]},
    {"op": ">", "args": [{"property": "collection.quantity"}, 1]}
  ]
}
```

Meaning: scoped Collection rows are in Simic Ramp Control and foil, and their summed quantity is greater than 1.

If a future workflow needs cards with copies in two separate locations, use two separate `withCollectionCard` predicates
in
an outer `and`.

## Tag Concept Search Rule

For deck-building concepts such as draw, ramp, recursion, sacrifice, removal, or protection, agents should normally:

1. Use `search_card_identity_tags` to resolve the intended Card Identity Tag.
2. Query with `hasTagInHierarchy(tag.id, resolvedTagId)`.

Direct `tag.slug`, `tag.label`, and `tag.alias` remain exact/direct queryables, but they are not fuzzy discovery and do
not
imply hierarchy-aware matching. Broad deck-building concepts should prefer hierarchy matching.

## Collection Quantity Semantics

`collection.quantity` is an aggregate over the active matching Collection row scope.

If a branch has Collection row predicates, `collection.quantity` sums rows that survived those predicates.

Example:

```json
{
  "op": "and",
  "args": [
    {"op": "=", "args": [{"property": "collection.locationName"}, "Simic Ramp Control"]},
    {"op": "=", "args": [{"property": "collection.finish"}, "foil"]},
    {"op": ">", "args": [{"property": "collection.quantity"}, 1]}
  ]
}
```

Meaning: the card has more than one foil copy in Simic Ramp Control.

When no Collection predicate has established a scope, `collection.quantity` means total owned quantity across the whole
Collection.

Example:

```json
{
  "filter": {"op": "contains", "args": [{"property": "identity.typeLine"}, "Planeswalker"]},
  "sortby": [{"property": "collection.quantity", "direction": "desc"}]
}
```

Meaning: find Planeswalker Card Identities and sort by total owned quantity; unowned matching cards sort as `0`.

Sorting never filters. If ownership is required, the query must explicitly include `collection.quantity > 0`.

## Branch-Local Quantity Semantics

`collection.quantity` is branch-local when it appears inside an `or` branch.

This should not match a card with one copy in each location:

```json
{
  "op": "or",
  "args": [
    {
      "op": "and",
      "args": [
        {"op": "=", "args": [{"property": "collection.locationName"}, "Simic Ramp Control"]},
        {"op": ">=", "args": [{"property": "collection.quantity"}, 2]}
      ]
    },
    {
      "op": "and",
      "args": [
        {"op": "=", "args": [{"property": "collection.locationName"}, "Main box"]},
        {"op": ">=", "args": [{"property": "collection.quantity"}, 2]}
      ]
    }
  ]
}
```

This should match that same card because the quantity applies after the union of the two location scopes:

```json
{
  "op": "and",
  "args": [
    {
      "op": "or",
      "args": [
        {"op": "=", "args": [{"property": "collection.locationName"}, "Simic Ramp Control"]},
        {"op": "=", "args": [{"property": "collection.locationName"}, "Main box"]}
      ]
    },
    {"op": ">=", "args": [{"property": "collection.quantity"}, 2]}
  ]
}
```

The SQL compiler cannot blindly flatten all Collection predicates into one `WHERE` plus one `HAVING`; it needs to
preserve
branch-shaped scopes where aggregates appear.

## Mixed `OR` Evidence Semantics

For mixed `or` queries, Collection evidence comes only from branches that actually matched through Collection
predicates.

Example:

```json
{
  "op": "or",
  "args": [
    {"op": "=", "args": [{"property": "collection.locationName"}, "Simic Ramp Control"]},
    {"op": "hasTagInHierarchy", "args": [{"property": "tag.id"}, "draw-tag-id"]}
  ]
}
```

If a card matches only through the draw tag branch, then:

- `include.collectionCards` returns an empty `collectionCards` array for that card.
- `sortby: collection.quantity` uses `0` for that card when the query contains Collection branch scope.
- Owned copies elsewhere should not be shown as if they caused the Collection branch to match.

If the user wants owned draw cards, the agent should express ownership directly:

```json
{
  "op": "and",
  "args": [
    {"op": ">", "args": [{"property": "collection.quantity"}, 0]},
    {"op": "hasTagInHierarchy", "args": [{"property": "tag.id"}, "draw-tag-id"]}
  ]
}
```

## SQL Query Shape

The primary SQL query should be identity-first: one row per Card Identity after grouping, sorting, and limiting.

Do not join tables unless required for the query. In particular:

- `identity.*` predicates query `card_identity` only.
- `legality.commander` predicates should use `exists` over `card_identity_format_legality`.
- Independent `tag.*` and `hasTagInHierarchy` predicates should use `exists` over `card_identity_tagging` and related
  tag
  tables.
- Collection predicates, Collection quantity sorting, and Collection quantity filtering may require joins through
  `card_printing`, `collection_card`, and `collection_location` or branch-shaped CTE/subquery scopes.
- One-to-one or many-to-one joins are acceptable when they do not multiply the Card Identity row set. One-to-many
  relationships should use `exists`, pre-aggregation, or bounded follow-up queries unless the one-to-many rows are the
  aggregate being intentionally grouped.

Avoid joining tags and Collection rows into the same aggregate query unless the SQL is explicitly guarded against row
multiplication. A card with two Collection rows and five tags can otherwise become ten joined rows and inflate
`sum(collection_card.quantity)`.

## Example SQL Shapes

Owned foil cards in a specific Collection Location with more than one scoped copy:

```sql
select
  ci.id,
  ci.name,
  sum(cc.quantity) as collection_quantity
from card_identity ci
join card_printing cp
  on cp.card_identity_id = ci.id
join collection_card cc
  on cc.card_printing_id = cp.id
join collection_location cl
  on cl.id = cc.collection_location_id
where cl.id = ?
  and cc.finish = 'foil'
group by ci.id
having sum(cc.quantity) > 1
order by ci.name asc
limit ?;
```

Owned cards with a direct `draw` tag, avoiding tag/Collection row multiplication:

```sql
select
  ci.id,
  ci.name,
  sum(cc.quantity) as collection_quantity
from card_identity ci
join card_printing cp
  on cp.card_identity_id = ci.id
join collection_card cc
  on cc.card_printing_id = cp.id
where exists (
  select 1
  from card_identity_tagging cit
  join card_identity_tag tag
    on tag.id = cit.tag_id
  where cit.card_identity_id = ci.id
    and tag.slug = 'draw'
)
group by ci.id
having sum(cc.quantity) > 0
order by ci.name asc
limit ?;
```

Hierarchy-aware owned cards for a resolved tag:

```sql
with recursive tag_scope(tag_id) as (
  select ?
  union
  select h.child_tag_id
  from card_identity_tag_hierarchy h
  join tag_scope scope
    on h.parent_tag_id = scope.tag_id
)
select
  ci.id,
  ci.name,
  sum(cc.quantity) as collection_quantity
from card_identity ci
join card_printing cp
  on cp.card_identity_id = ci.id
join collection_card cc
  on cc.card_printing_id = cp.id
where exists (
  select 1
  from card_identity_tagging cit
  where cit.card_identity_id = ci.id
    and cit.tag_id in (select tag_id from tag_scope)
)
group by ci.id
having sum(cc.quantity) > 0
order by ci.name asc
limit ?;
```

Strong draw using explicit same-tagging scope:

```sql
with recursive tag_scope(tag_id) as (
  select ?
  union
  select h.child_tag_id
  from card_identity_tag_hierarchy h
  join tag_scope scope
    on h.parent_tag_id = scope.tag_id
)
select
  ci.id,
  ci.name
from card_identity ci
where exists (
  select 1
  from card_identity_tagging cit
  where cit.card_identity_id = ci.id
    and cit.tag_id in (select tag_id from tag_scope)
    and cit.weight = 'strong'
)
order by ci.id asc
limit ?;
```

## Recursive CTE Decision

Use recursive CTEs for `hasTagInHierarchy` filtering. This keeps hierarchy-aware matching in SQL without adding a
closure
table or keeping hierarchy expansion in TypeScript for query execution.

`include.tags` projection is different: hydrate direct taggings for the final IDs and compute inherited tag projections
in
bounded TypeScript. Do not force inherited projection into the primary SQL query.

## Include Hydration

The primary query should return final Card Identity IDs and scalar fields in result order. Includes should be hydrated
by
bounded follow-up queries for only those final IDs.

- `include.collectionCards`: hydrate Collection Card rows for final IDs. If Collection predicates are present, return
  only
  rows that contributed to matching Collection branches. If no Collection predicates are present, return all owned rows
  for
  each returned Card Identity.
- `include.legalities`: hydrate requested format legalities for final IDs only.
- `include.tags`: hydrate direct taggings, tag definitions, aliases, and hierarchy data needed to produce deterministic
  `tags.direct` and `tags.inherits` for final IDs only.

Hydration must preserve the primary result ordering. Do not rely on SQL `IN (...)` returning rows in the same order as
the
input ID list; reorder in TypeScript if needed.

## Sorting Semantics

- Omitted `sortby` defaults to `identity.id asc`.
- Explicit sorting should append a deterministic `identity.id asc` tie-breaker unless already sorting by `identity.id`.
- Nullable reference sorts are null-last in both ascending and descending directions.
- `collection.quantity` sorting uses the matching Collection row scope when one exists.
- If a mixed-branch query returns an identity through only a non-Collection branch, its scoped Collection quantity sort
  value is `0`.
- If no Collection predicate exists anywhere in the filter, `collection.quantity` sorting means total owned quantity
  across
  the whole Collection, with unowned matches as `0`.

## SQL Compiler Shape

Use a small internal SQL fragment/AST model rather than ad hoc string concatenation. Keep it inside `packages/sqlite`,
likely split out of `card-query-repository.ts` if it grows beyond trivial size.

Suggested shape:

```ts
type SqlFragment = {
  readonly text: string;
  readonly params: readonly unknown[];
};
```

The exact type can differ, but the requirements are:

- Every user/query value is a bound parameter.
- No user-controlled value is interpolated into SQL text.
- Table names, column names, operators, sort directions, and CTE names are emitted only from fixed allowlist mappings.
- The compiler should be safe from SQL injection by construction.
- Validation should reject invalid operator/property combinations before SQL execution.

## Validation Changes

Add `withTagging` and `withCollectionCard` to the validated operator set.

Validation rules:

- Each accepts exactly one child filter.
- `withTagging` child subtrees may contain only allowed tag predicates and boolean groups over them.
- `withCollectionCard` child subtrees may contain only allowed Collection predicates and boolean groups over them.
- `not` remains unsupported over relationship-backed predicates. Do not allow `not` inside these operators.
- Keep strict envelope validation: unknown fields, invalid operators, invalid queryables, invalid value types, empty
  boolean
  groups, empty sort/include arrays, and duplicate sort/include fields should return validation errors.

## Agent Skill Update

The `.opencode/skills/query-cards/SKILL.md` file must be updated as part of the public language slice, before agents
rely
on the new semantics.

The skill should teach:

- Use `search_card_identity_tags` first for concepts.
- Use `hasTagInHierarchy` for normal concept matching.
- Use independent `hasTagInHierarchy` predicates for multi-tag searches such as draw + ramp.
- Use `withTagging` when tag metadata must apply to the same tagging row, such as strong draw.
- Use simple Collection predicates for normal owned-row searches.
- Use `withCollectionCard` for explicit Collection row-scope grouping in complex cases.
- `include.collectionCards` is match evidence, not automatically all owned copies in mixed `or` queries.
- `sortby: collection.quantity` sorts by matching scoped quantity where applicable and does not filter unless ownership
  is
  explicit.

## Test Strategy

Use temp migrated SQLite databases and explicit fixtures. Do not test against `.data/mtg-agent.sqlite`.

Prefer observable repository behavior tests over full SQL string snapshots. Small SQL compiler unit tests are acceptable
for parameter binding or fragment composition, but the main contract is Card Query behavior.

Avoid exact timing assertions. Performance verification should come from SQL pushdown structure and optional developer
inspection, not brittle wall-clock tests.

Required test cases:

1. Public language validation accepts `withTagging` and `withCollectionCard` with valid relationship-only subtrees.
2. Validation rejects identity/legal/Collection predicates inside `withTagging`.
3. Validation rejects identity/legal/tag predicates inside `withCollectionCard`.
4. Multi-tag search matches a card that has draw and ramp as separate taggings.
5. Strong draw through `withTagging` does not match a card with weak draw plus unrelated strong ramp.
6. `hasTagInHierarchy` matches descendant taggings through recursive hierarchy.
7. Owned cards with a tag do not inflate `collection.quantity` because of tag row multiplication.
8. Collection same-row shorthand: location + finish must match the same Collection row scope.
9. Collection quantity aggregates only the rows surviving Collection predicates.
10. Branch-local Collection quantity under `or` does not combine quantities across separate branches.
11. Quantity after an `or` union scope does combine the unioned rows when the quantity predicate is outside the `or`.
12. Mixed `or` include evidence: tag-only branch match returns empty `collectionCards` when the Collection branch did
    not
    match.
13. Mixed `or` quantity sorting uses `0` for identities with no retained Collection branch evidence.
14. No Collection predicate plus `sortby: collection.quantity` sorts by total owned quantity.
15. `include.legalities`, `include.tags`, and `include.collectionCards` hydrate only requested includes and preserve
    primary
    result order.
16. Null-last sorting for nullable reference properties still holds.

## Implementation Slices

### Slice 1: Public Language and Documentation

- Update core Card Query types and validation for `withTagging` and `withCollectionCard`.
- Add parser/validation tests for the new operators and relationship-only subtree restrictions.
- Update `docs/plans/card-query.md` to reference this plan and summarize the explicit relationship-scope operators.
- Update `.opencode/skills/query-cards/SKILL.md` with practical agent guidance and examples.

### Slice 2: Semantic Repository Tests

- Add SQLite repository tests using temp migrated databases and explicit fixture data for the required cases above.
- These tests should encode the intended SQL-backed semantics, not old evaluator quirks.
- If useful during development, the old evaluator may be used as a temporary test oracle for non-ambiguous existing
  cases,
  but it must not remain as production fallback.

### Slice 3: SQL-backed Repository Replacement

- Replace the current in-memory repository implementation in `packages/sqlite/src/card-query-repository.ts`.
- Introduce an internal SQL compiler module if needed.
- Execute the primary identity-first SQL query with bound parameters.
- Hydrate requested includes with bounded follow-up queries for final IDs.
- Remove broad table materialization from query execution.
- Ensure all supported documented features compile to SQL-backed behavior.

### Slice 4: Verification

- Run targeted SQLite repository tests.
- Run core parser tests.
- Run `bun run typecheck`.
- Optionally run full `bun test` if practical.
- Optionally inspect `EXPLAIN QUERY PLAN` manually against developer data for representative queries, but do not require
  local `.data` for automated tests.

## Open Follow-Ups After Replacement

- Consider explicit pagination only if real workflows need it. Current first slice remains `limit` only.
- Consider full-text search or trigram-like indexing for substring `contains` if name/type/oracle text search remains
  slow.
  Normal B-tree indexes do not help much for leading-wildcard substring searches.
- Consider explicit anti-existence operators later if agents need "unowned," "not in location," or "without tag"
  queries.
  Do not overload `!=` for relationship-backed predicates.
- Consider relationship-scope operators for future relationship families only when a real workflow needs them.
