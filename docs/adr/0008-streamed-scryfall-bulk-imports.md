# Streamed Scryfall bulk imports

Scryfall `all_cards` bulk data is large enough that both full raw JSON parsing and accumulating all mapped records in core memory create unnecessary memory pressure. Core owns streaming JSON parsing from `ReadableStream<Uint8Array>` sources and lazily validates/maps records into repository imports; SQLite consumes those mapped records into transaction-scoped staging tables before replacing live Card Identity and Card Printing reference data. We chose staged replacement over direct streamed writes into target tables because it preserves failed-import non-destructiveness while allowing set-based validation and clearer diagnostics before live reference data is replaced; hot staging loops may use lower-level prepared SQLite statements while repository ports continue to hide persistence-specific types from core.
